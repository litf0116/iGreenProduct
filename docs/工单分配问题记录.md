# 工单分配功能问题记录

## 问题描述
在 `/tickets` 页面创建工单时，"assign to"（分配给）选择功能存在业务逻辑不明确的问题。

## 问题详情

### 当前实现状态
根据代码分析，发现了以下情况：

**1. 数据库设计**
- `tickets` 表中 `assigned_to` 字段为 `VARCHAR(36)`
- 设置了外键约束：`FOREIGN KEY (assigned_to) REFERENCES users(id)`
- 这表明该字段应该存储用户ID

**2. 前端界面实现**
在 `igreen-front/src/components/CreateTicket.tsx` 中：
```typescript
// 第 310-325 行：分配给字段
<Label>{t("assignTo")}</Label>
<Select value={assignedTo} onValueChange={setAssignedTo} required>
  <SelectTrigger>
    <SelectValue placeholder={t("assignTo")} />
  </SelectTrigger>
  <SelectContent>
    {groups
      .filter((g) => g.status === "active")
      .map((group) => (
        <SelectItem key={group.id} value={group.id}>
          {group.name}
        </SelectItem>
      ))}
  </SelectContent>
</Select>
```

**问题关键**：
- 界面显示的是用户组（groups）选项
- 但实际存储到数据库的是 group.id
- 而数据库外键约束期望的是 user.id

### 冲突分析

**数据库约束**：
```sql
assigned_to VARCHAR(36),
FOREIGN KEY (assigned_to) REFERENCES users(id)
```

**前端实现**：
- 选择的是 `groups`（用户组）
- 存储的是 `group.id`
- 但数据库期望的是 `users.id`

**这会导致**：
1. 数据库外键约束错误
2. 工单分配逻辑不清晰
3. 后续查询和显示可能出现问题

## 需要确认的问题

**核心问题**：工单应该分配给用户组还是个人用户？

### 方案 A：分配给用户组
- 修改数据库约束，允许 `assigned_to` 引用 `groups.id`
- 工单分配给整个用户组，组内成员可以抢单或由管理员二次分配
- 适合团队协作模式

### 方案 B：分配给个人用户
- 修改前端界面，显示用户列表而非用户组列表
- 工单直接分配给具体的工程师
- 适合明确责任到人的模式

### 方案 C：两级分配
- 第一级：分配给用户组
- 第二级：组内再分配给具体用户
- 需要额外的状态管理和流程设计

## 业务影响

这个问题影响：
1. **工单创建流程** - 当前可能无法正常保存工单
2. **工单分配逻辑** - 分配机制不明确
3. **权限管理** - 用户权限检查可能出错
4. **统计报表** - 工单统计和查询可能异常

## 建议

**紧急处理**：需要与客户/产品经理确认业务需求，明确工单分配的具体流程：
1. 是否需要支持分配给用户组？
2. 是否需要支持分配给个人用户？
3. 分配后的工单流转机制如何设计？

**技术处理**：根据业务确认结果，选择上述方案之一进行实现。
